Craft Atlas · 04 · User Experience

The architecture of where am I, and where next?

UX is not decoration applied at the end. It is the structure underneath — how a site is divided, named, navigated, and recovered from when things break. Mined from 80 live teardowns, here is what elite sites actually do.

01 — FoundationInformation architecture

Before a single pixel of style, every great site answers one question: what are the chunks, and in what order do they reveal? Across our teardowns, the winning IA is almost always a linear scroll narrative built from self-contained bands — not a sprawling sitemap you have to hold in your head.

Linear ↗ structures its entire homepage as a numbered chapter system: hero → logo wall → thesis → five numbered capability sections (1.0–5.0), each with sub-feature tabs (1.1 Agent, 1.2 Triage…) → changelog → quotes → CTA → mega-footer. The numbering is the IA — it tells you exactly how deep you are and how much is left. Stripe ↗ runs a different but equally legible spine: a top-of-funnel ladder — hero claim → social-proof logo wall → 6-proposition accordion → stat band (135+ currencies, $1.9T volume, 99.999% uptime) → segmented Enterprise/Startups/Platforms tabs → developer section → news carousel → dual CTA → mega-footer.

Three IA principles recur across the reference set:

One idea per band

Apple ↗ uses full-bleed stacked bands with deliberately low density — "one idea per screen." The page is long but never crowded; whitespace separates concepts so nothing competes.

Semantic heading map

Apple ↗ reserves h1 for the brand anchor, h2 for marquee products, h3 for secondary tiles — a clean scannable outline a screen reader can traverse top-to-bottom.

Dense only where expected

Linear ↗ keeps marketing copy sparse but makes the embedded app mockups intentionally dense — density signals product depth, and the user opts into it.

The mega-footer is your sitemap, not your nav. Nearly every reference site — Linear ↗, Stripe ↗, Apple ↗, Arc ↗ — keeps the top nav lean and dumps full breadth into a dense multi-column footer. The header is for the 3–5 things most people want; the footer catches everything else without cluttering the chrome.

02 — MovementNavigation patterns

A navigation bar is a promise about scope. The teardowns cluster into three honest archetypes — and the best sites pick one deliberately rather than defaulting to a hamburger.

Pattern A — Lean text nav (brochure confidence)

Things ↗ ships the most confident nav we found: wordmark + three text links (Features, Support, Blog). No hamburger, no mega-menu, no sticky-on-scroll transformation. Critically, the download CTAs are deliberately not in the nav — they live in a dedicated mid-page "Get Things" section, keeping the chrome uncluttered. The nav assumes you came to learn, get help, or read, and pushes the purchase decision into the body. Sara Soueidan ↗ does the same with honest underlined text links in a real <nav aria-label="Main"> — no hamburger on desktop, no mega-menu.

Pattern B — Mega-menu (when breadth is real)

When you genuinely have dozens of destinations, use categorized panels — but make them real ARIA buttons, not hover traps. Stripe ↗, Vercel ↗, Apple ↗, Linear ↗ and Family ↗ all expose Products/Resources/Solutions as aria-expanded buttons. The Stripe nav is the famous one: a single morphing dropdown panel that tweens its width/height/position between menu items rather than swapping four discrete dropdowns — moving Products→Solutions feels like one fluid surface.

Click a trigger — the panel morphs open in place instead of popping a separate menu (the Stripe move). The triggers are real aria-expanded buttons you can Tab to.

Family ↗ sharpens this: dropdowns are content-rich (ConnectKit, Family Account, Blog, Changelog, FAQs) with a title and a one-line description per item — "mega-menu-lite." The menu becomes a tiny landing page.

Pattern C — Sticky cross-sell / contextual ribbon

Arc ↗ pins a "Meet Dia" promo strip directly under the nav that rides along on scroll — keeping a cross-sell CTA permanently visible without a modal. Vercel ↗ does the restrained version: an event ribbon styled as a contextual chip ("Ship 26 is coming…"), not an intrusive bar. The lesson: persistent promotion is fine if it's a thin companion to the nav — not an interstitial that blocks the page.

/* The transparent-over-hero sticky nav (Linear, Vercel pattern).
   Backdrop-blur appears only once you scroll past the hero. */
.site-header {
  position: sticky;
  top: 0;
  background: transparent;
  transition: background .25s ease, border-color .25s ease;
}
.site-header[data-scrolled] {
  background: rgba(11, 12, 14, .72);
  backdrop-filter: saturate(150%) blur(14px);
  border-bottom: 1px solid var(--line);
}
// Toggle the scrolled state with a sentinel + IntersectionObserver
// (cheaper and jank-free vs. a scroll listener).
const header = document.querySelector('.site-header');
const sentinel = document.querySelector('#hero-sentinel');

new IntersectionObserver(([entry]) => {
  header.toggleAttribute('data-scrolled', !entry.isIntersecting);
}, { rootMargin: '-1px 0px 0px 0px', threshold: 0 }).observe(sentinel);

Anti-pattern: the hover-only mega-menu. A dropdown that opens on hover with no keyboard or click path strands touch and keyboard users entirely. Every reference site uses aria-expanded buttons you can Tab to and Enter on. If it only works with a mouse, it's broken for a large slice of your visitors.

03 — OrientationWayfinding & orientation

Wayfinding answers "where am I?" mid-scroll. The strongest sites bake orientation into the content rather than relying on a breadcrumb trail.

  • Numbered chapters — Linear ↗ uses 1.0–5.0 capability sections so position is always explicit; wayfinding "comes from the numbered chapter system rather than nav anchors."
  • Eyebrow labels — Josh Comeau ↗ introduces every section with a small uppercase eyebrow (ARTICLES AND TUTORIALS, BROWSE BY CATEGORY, POPULAR CONTENT). Family ↗ puts a green "Simple" pill above headlines to categorize. The eyebrow is a cheap, scannable "you are here" marker.
  • A fixed identity rail — Brittany Chiang ↗'s signature split-screen pins name, role, and in-page nav in a sticky left column while content scrolls past on the right. The identity never leaves the viewport, so you always know whose site you're on and what the sections are.
  • An anchored focal point — Active Theory ↗ has almost no DOM content; wayfinding "relies on motion and the persistent center orb as an anchor rather than breadcrumbs."

That active-section highlight is a scroll-spy. The robust, broadly-supported version is a tiny IntersectionObserver:

// Scroll-spy: mark the in-view section's nav link as current.
const links = new Map(
  [...document.querySelectorAll('.rail-nav a')]
    .map(a => [a.getAttribute('href').slice(1), a])
);

const spy = new IntersectionObserver((entries) => {
  for (const e of entries) {
    if (!e.isIntersecting) continue;
    links.forEach(a => a.removeAttribute('aria-current'));
    links.get(e.target.id)?.setAttribute('aria-current', 'true');
  }
}, { rootMargin: '-45% 0px -45% 0px' });  // trigger near viewport center

document.querySelectorAll('main section[id]').forEach(s => spy.observe(s));

Pair scroll-spy with scroll-margin-top on your headings (this page sets 5.5rem) so anchor jumps land below the sticky nav, not hidden behind it. This one property is the difference between a polished and a broken anchor-link experience.

04 — ArrivalFirst impression & onboarding

You get one frame to establish what kind of place this is. The teardowns show two opposite — both valid — strategies.

Strategy 1 — Instant content (credibility through speed)

Sara Soueidan ↗ loads instantly with a 100% system-font stack — zero web-font requests, zero FOUT, "perfect rendering everywhere." For a content site, the fastest possible first paint is the first impression: it says "I respect your time." Things ↗ and Apple ↗ similarly front-load the single most important idea above the fold and let you scroll into depth.

Strategy 2 — The crafted preloader (personality moment)

For experiential sites, a loader buys asset-load time and sets tone. Dennis Snellenberg ↗ cycles a multilingual greeting sequence ("Hello / Bonjour / स्वागत हे / Ciao / Olá…") — nine h2s that double as a loading counter and a personality moment. Active Theory ↗ makes the brand orb itself the loading/idle centerpiece, so the very first frame is already the experience.

Pitfall: the loader that's slower than the content. A preloader is only justified if real work is happening behind it (WebGL scene compile, heavy media). On a content site, an artificial 2-second intro is a tax you charge every returning visitor. Gate it behind prefers-reduced-motion and skip it on repeat loads.

// Honest preloader: show it only while assets actually load,
// respect motion prefs, and never block repeat visits.
const seen = sessionStorage.getItem('intro-seen');
const reduced = matchMedia('(prefers-reduced-motion: reduce)').matches;

if (seen || reduced) {
  document.documentElement.dataset.intro = 'skip';
} else {
  await Promise.all([
    document.fonts.ready,
    ...heavyAssets.map(load),   // your real load promises
  ]);
  sessionStorage.setItem('intro-seen', '1');
  document.documentElement.dataset.intro = 'done';
}

05 — VoiceContent design

Content design is UX — the words are the interface. The teardowns reward two moves repeatedly.

Two-tone emphasis instead of a second font. Linear ↗ renders the first clause of a heading in bright white and the continuation in muted gray within the same h2 ("A new species of product tool." bright, the rest dimmed) — emphasis without a second typeface or hue. It directs the eye to the claim and demotes the supporting clause in one element.

<!-- One heading, two emphasis levels — no extra font, no color hue -->
<h2 class="lede-heading">
  <span class="hot">A new species of product tool.</span>
  Built for how software is designed and shipped today.
</h2>

<style>
  .lede-heading { color: var(--ink-dim); }   /* continuation dims back */
  .lede-heading .hot { color: var(--ink); }  /* claim stays bright     */
</style>

Descriptions that teach inside navigation. Family ↗ and Stripe ↗ never ship a bare link list — each destination gets a one-line description so the menu itself answers "is this the thing I want?" before the click. This is content design doing wayfinding's job.

Write labels for strangers, not insiders. Things ↗ labels its nav "Features / Support / Blog" — verbs of intent, not internal product names. The test: a first-time visitor should predict what's behind every nav item without clicking. If a label needs a tooltip to be understood, it's the wrong label.

06 — SpeedPerformance as UX

Speed is not a separate engineering concern — it is felt as quality. Slow is a UX bug. The teardowns treat performance as a design constraint.

  • Self-hosted variable fonts. Vercel ↗ ships one variable family (Geist Sans + Geist Mono) as self-hosted woff2 — one typeface covers display and mono via weight + tracking, cutting requests. Linear ↗ does the same with Inter Variable.
  • The zero-font extreme. Sara Soueidan ↗ loads no web fonts at all — a deliberate "performance + resilience decision (zero font requests, zero FOUT, perfect rendering everywhere)."
  • Lazy-load below-the-fold media. Dennis Snellenberg ↗ ships vanilla-lazyload so off-screen imagery never competes with the first paint.
  • GPU-only motion. Per Vercel ↗'s own published guidelines: motion priority is CSS > Web Animations API > JS libs; animate only transform/opacity, never layout-triggering properties, no gratuitous autoplay.
<!-- Below-the-fold images: native lazy + explicit dimensions kill CLS -->
<img src="case-study.webp" alt="…"
     loading="lazy" decoding="async"
     width="1200" height="675" />

<!-- The ONE hero image gets the opposite treatment: load it eagerly -->
<img src="hero.webp" alt="…" fetchpriority="high" />

Always set width/height (or aspect-ratio) on media. The reserved box prevents Cumulative Layout Shift — content jumping as images arrive is the most common, most jarring perceived-performance failure, and it's free to fix.

07 — BaselineAccessibility is the baseline

The reference-class sites treat a11y as structure, not a plugin. The clearest tell is the skip link as the literal first DOM node.

Linear ↗ includes an accessible "Skip to content" link as the first focusable element ("good a11y discipline"). Sara Soueidan ↗ goes furthest: a skip link is "the very first element in the DOM," followed by real ARIA-labelled landmarks — "accessibility baked into structure, not bolted on." Vercel ↗ ships a skip-to-content link too.

Skip to content

Click into this box, then press Tab — the skip link slides in from off-screen. It's invisible until focused, then it's the first stop for keyboard users.

<body>
  <a class="skip-link" href="#main">Skip to content</a>
  <header>…nav…</header>
  <main id="main">…</main>
</body>

<style>
  .skip-link {
    position: absolute; left: .5rem; top: .5rem;
    transform: translateY(-200%);   /* off-screen but focusable */
    transition: transform .15s ease;
  }
  .skip-link:focus { transform: translateY(0); }  /* reveal on focus */
</style>

Beyond the skip link, the patterns that recur:

  • Never use color as the only signifier. Sara Soueidan ↗ keeps links black, differentiated by underline not hue — "so color is never the sole signifier."
  • Real landmarks. Every nav is a labelled <nav aria-label="…">; sections are regions with ARIA headings (Brittany Chiang ↗).
  • Visible focus + keyboard paths. focus-visible outlines and keyboard-operable skip / back-to-top anchors (Sara Soueidan ↗) — "microinteractions serve accessibility, not spectacle."
  • Respect prefers-reduced-motion. This page disables animation under the query; honor it for every loader and scroll effect.

Pitfall: ARIA theater. Sprinkling role and aria-* onto <div>s does not make a site accessible — and often makes it worse. The reference sites use real semantic elements (nav, main, button, headings) first, and add ARIA only to fill genuine gaps. A native <button> beats <div role="button"> every time.

08 — EdgesError & empty states

The forgotten screens — empty lists, failed loads, 404s, first-run states — are where polish is proven. The teardowns show the discipline of states that carry their own way out.

Stripe ↗'s prev/next carousel controls ship "proper disabled states on the first slide" — the control communicates its own boundary instead of failing silently. The same principle scales to every dead-end: an empty or error state should explain what, why, and the next action.

Every dead end keeps the chrome. A 404 or error page that strips the nav traps the user. Keep the header and footer sitemap on error screens — the Arc ↗ / Linear ↗ mega-footer is a built-in escape hatch from any broken state. The cheapest error recovery is the navigation you already built.

09 — ShipThe steal-this checklist

A scannable pass to run against any page before you ship it.

Architecture

  • One idea per band (Apple)
  • Clean h1→h2→h3 outline (Apple)
  • Breadth lives in the footer, not the nav
  • Density only where users opt in (Linear mockups)

Navigation & wayfinding

  • Pick one nav archetype on purpose (Things vs Stripe)
  • aria-expanded buttons, not hover traps
  • Eyebrow labels or numbered chapters for position
  • Scroll-spy aria-current + scroll-margin-top

Performance & a11y

  • One self-hosted variable font (Vercel / Linear)
  • Skip link as the first DOM node (Sara Soueidan)
  • width/height on every image (no CLS)
  • Honor prefers-reduced-motion on loaders

States

  • Empty = icon + reframed line + one CTA
  • Disabled controls show their boundary (Stripe)
  • 404/error keeps the nav + footer escape hatch
  • Loaders are honest, skippable, reduced-motion-safe
